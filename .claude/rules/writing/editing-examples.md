# 編集実例リファレンス

## 目的

記事編集時の具体的なbefore/after例を提供し、望ましい編集パターンを示す。見出しの改善、文章の簡潔化、AI表現の削除、技術的補足の追加などの実例を含む。

## 実際の編集実例

<!-- textlint-disable -->

### 見出しの改善例

#### パターン1: 冗長な見出しの簡潔化

```markdown
<!-- 修正前 -->

## なぜURLに認証情報を埋め込むのか

<!-- 修正後 -->

## 課題
```

**理由**: 「なぜ〜か」という疑問形は冗長。セクション名を名詞形にして簡潔に。

#### パターン2: 長い見出しの分割

```markdown
<!-- 修正前 -->

## 使用上の注意と推奨場面

<!-- 修正後 -->

## 注意点

## 適用場面
```

**理由**: 複数の内容を1つの見出しに詰め込まない。セクションを分離して構造を明確に。

#### パターン3: 技術用語の統一

```markdown
<!-- 修正前 -->

## Git URLについて

<!-- 修正後 -->

## Git URL形式
```

**理由**: 「について」は曖昧。具体的な内容を示す名詞で表現。

### 文章の簡潔化例

#### パターン1: AI特有の説明表現の削除

```markdown
<!-- 修正前 -->

この章では、URLに認証情報を埋め込む方法について説明する。

<!-- 修正後 -->

URLに認証情報を埋め込む方法は以下の通りである。
```

**理由**: 「〜について説明する」は冗長。直接的な表現に変更。

#### パターン2: 冗長な「できる」の削除

```markdown
<!-- 修正前 -->

この方法を利用することで、CI/CDパイプラインで認証を自動化できる。

<!-- 修正後 -->

この方法を利用すれば、CI/CDパイプラインで認証を自動化できる。
```

**理由**: 「〜することで」は冗長。「〜すれば」で簡潔に。

#### パターン3: 回りくどい表現の短縮

```markdown
<!-- 修正前 -->

セキュリティを確保することが重要である。

<!-- 修正後 -->

セキュリティの確保が重要である。
```

**理由**: 「〜することが」は冗長。名詞化して簡潔に。

#### パターン4: 不要な修飾語の削除

```markdown
<!-- 修正前 -->

非常に便利な方法である。

<!-- 修正後 -->

便利な方法である。
```

**理由**: 「非常に」「とても」などの感情的な修飾語は削除。

### AI表現の削除例

#### パターン1: 「〜と言える」の削除

```markdown
<!-- 修正前 -->

この方法は効率的だと言える。

<!-- 修正後 -->

この方法は効率的である。
```

**理由**: 「〜と言える」は不要。断定形で表現。

#### パターン2: 「〜に関して」「〜に対して」の簡略化

```markdown
<!-- 修正前 -->

セキュリティに関して注意が必要である。

<!-- 修正後 -->

セキュリティには注意が必要である。
```

**理由**: 「〜に関して」は冗長。「〜には」で簡潔に。

#### パターン3: 「〜を紹介する」の削除

```markdown
<!-- 修正前 -->

以下では、具体的な実装例を紹介する。

<!-- 修正後 -->

具体的な実装例は以下の通りである。
```

**理由**: 「紹介する」は冗長。直接的な表現に変更。

### 技術的補足の追加例

#### パターン1: 用語の説明追加

```markdown
<!-- 修正前 -->

userinfo部分に認証情報を含める。

<!-- 修正後 -->

userinfo（URI仕様RFC 3986で定義された`username:password`形式の部分）に認証情報を含める。
```

**理由**: 技術用語の初出時は説明を追加。

#### パターン2: セキュリティ注意事項の追加

```markdown
<!-- 修正前 -->

URLに認証情報を埋め込む。

<!-- 修正後 -->

URLに認証情報を埋め込む。ただし、この方法はセキュリティリスクがあるため、ログ記録や履歴に注意が必要である。
```

**理由**: セキュリティに関わる内容には必ず注意事項を追加。

#### パターン3: 制約条件の明示

```markdown
<!-- 修正前 -->

この方法はCI/CDパイプラインで利用できる。

<!-- 修正後 -->

この方法はCI/CDパイプラインで利用できる。ただし、本番環境では推奨されない。
```

**理由**: 適用場面の制限を明確に記載。

### 構成の改善例

#### パターン1: セクションの分離

````markdown
<!-- 修正前 -->

## 実装方法

URLに認証情報を埋め込む方法は以下の通りである。

\```bash
git clone https://username:password@github.com/user/repo.git
\```

ただし、セキュリティリスクがあるため注意が必要である。

<!-- 修正後 -->

## 実装方法

URLに認証情報を埋め込む方法は以下の通りである。

\```bash
git clone https://username:password@github.com/user/repo.git
\```

## 注意点

セキュリティリスクがあるため、以下に注意が必要である。
````

**理由**: 注意点は独立したセクションに分離。

#### パターン2: 段階的な説明の追加

````markdown
<!-- 修正前 -->

## 実装

\```javascript
const result = doSomething();
\```

<!-- 修正後 -->

## 実装

### 基本的な使い方

\```javascript
const result = doSomething();
\```

### 応用例

\```javascript
const result = doSomethingAdvanced();
\```
````

**理由**: 基本→応用の順で段階的に説明。

### コード例の改善例

#### パターン1: 言語指定の追加

````markdown
<!-- 修正前 -->

```
git clone https://username:password@github.com/user/repo.git
```

<!-- 修正後 -->

```bash
git clone https://username:password@github.com/user/repo.git
```
````

**理由**: コードブロックには必ず言語指定を追加。

#### パターン2: コメントの追加

````markdown
<!-- 修正前 -->

```javascript
const result = doSomething();
processData(result);
```

<!-- 修正後 -->

```javascript
// データを取得
const result = doSomething();

// 取得したデータを処理
processData(result);
```
````

**理由**: 重要な箇所にはコメントを追加して意図を明確に。

#### パターン3: 比較パターンの追加

````markdown
<!-- 修正前 -->

```javascript
const data = getData() ?? defaultData;
```

<!-- 修正後 -->

従来の方法:

```javascript
const data = getData();
if (data === null || data === undefined) {
  data = defaultData;
}
```

改善後:

```javascript
const data = getData() ?? defaultData;
```
````

**理由**: 従来の方法と比較して改善点を明確に。

### 文体の統一例

#### パターン1: 「です・ます」から「である」への変換

```markdown
<!-- 修正前 -->

この方法は便利です。

<!-- 修正後 -->

この方法は便利である。
```

**理由**: 技術記事は「である調」で統一。

#### パターン2: 口語表現の削除

```markdown
<!-- 修正前 -->

この方法はちょっと複雑だけど便利な感じである。

<!-- 修正後 -->

この方法は複雑だが便利である。
```

**理由**: 「ちょっと」「〜な感じ」などの口語表現を削除。

**理由**: リンクには説明を追加して可読性を向上。

## 編集の優先順位

1. **簡潔性** - 無駄な文章の徹底的な削除
2. **実用性** - 自分が使うための背景情報は保持
3. **正確性** - 技術的事実の徹底検証
4. **読みやすさ** - 自然で分かりやすい表現

## 編集プロセス

1. **内容の簡潔化** - 冗長表現・AI的表現の削除
2. **構成の最適化** - 見出し・章立ての改善
3. **事実確認** - 外部資料によるファクトチェック
4. **表現の調整** - 文章表現・textlint対応

## 編集時の注意事項

基本的な制約については`CLAUDE.md`の「必須遵守事項」を参照。以下は編集時の追加注意事項：

<!-- textlint-disable -->

- 引用・例文として意図的に残された「です・ます」調は変更しない
<!-- textlint-enable -->
- コードブロックには必ず言語指定を追加
- 画像パスは必ず`/images/`で開始
