# 編集実例リファレンス

記事編集時の具体的なbefore/after例を提供する。

**適用ルール**: 各カテゴリ （見出し / AI表現 / 簡潔化 / 補足 / 構成 / コード / リンク） は、ドラフトに該当する問題箇所がある場合にのみ適用する。該当箇所がないカテゴリは適用対象外とし、例を無理に追加しない。

## 見出しの改善

### 冗長な見出しの簡潔化

```markdown
<!-- 修正前 -->

## なぜURLに認証情報を埋め込むのか

<!-- 修正後 -->

## 課題
```

**理由**: 疑問形は冗長。名詞形で簡潔に。

### 長い見出しの分割

```markdown
<!-- 修正前 -->

## 使用上の注意と推奨場面

<!-- 修正後 -->

## 注意点

## 適用場面
```

**理由**: 複数の内容を1つの見出しに詰め込まない。

### 曖昧な見出しの具体化

```markdown
<!-- 修正前 -->

## Git URLについて

<!-- 修正後 -->

## Git URL形式
```

**理由**: 「について」は曖昧。具体的な名詞で表現。

## 文章の簡潔化

### 冗長な接続表現

`〜ことで` は手段や条件を示す場面で冗長になりやすい。ただし機械的に置換しない: 文脈で `〜ことで` のほうが自然なとき (動作の連続や並列を示すとき等) は保持してよい。

```markdown
<!-- 修正前 -->

この方法を利用することで、CI/CDパイプラインで認証を自動化できる。

<!-- 修正後 -->

この方法を利用すれば、CI/CDパイプラインで認証を自動化できる。
```

`まず` `次に` のような順序接続詞は、箇条書きや手順番号があるときは冗長になる。文章が時系列で自明な場合も削除する。

```markdown
<!-- 修正前 -->

まず、環境変数を設定する。次に、以下のコマンドを実行する。

<!-- 修正後 -->

環境変数を設定する。以下のコマンドを実行する。
```

### 不要な強調語

`非常に` `とても` `かなり` `極めて` のような強調語は技術記事では冗長になりやすい。事実だけで十分伝わる場合は削除する。ベンチマーク数値等、根拠のある強調は保持してよい。

```markdown
<!-- 修正前 -->

この方法は非常に便利である。

<!-- 修正後 -->

この方法は有効である。
```

### 回りくどい名詞化

```markdown
<!-- 修正前 -->

セキュリティを確保することが重要である。

<!-- 修正後 -->

セキュリティの確保が重要である。
```

### 冗長な前置詞表現

```markdown
<!-- 修正前 -->

セキュリティに関して注意が必要である。

<!-- 修正後 -->

セキュリティには注意が必要である。
```

## 技術的補足の追加

### 用語の説明

```markdown
<!-- 修正前 -->

userinfo部分に認証情報を含める。

<!-- 修正後 -->

userinfo（URI仕様RFC 3986で定義された`username:password`形式の部分）に認証情報を含める。
```

**理由**: 技術用語の初出時は説明を追加。

### セキュリティ注意事項

**適用場面**: リスクや副作用の警告が必要な技術 (認証情報の扱い、権限昇格、データ漏洩リスク等)。

```markdown
<!-- 修正前 -->

URLに認証情報を埋め込む。

<!-- 修正後 -->

URLに認証情報を埋め込む。ただし、この方法はセキュリティリスクがあるため、ログ記録や履歴に注意が必要である。
```

### 制約条件の明示

**適用場面**: 適用範囲や推奨場面を限定したいとき (環境依存、非推奨ケースの明示等)。セキュリティ注意事項と区別する: セキュリティはリスクの警告、制約条件は適用場面の限定。

```markdown
<!-- 修正前 -->

この方法はCI/CDパイプラインで利用できる。

<!-- 修正後 -->

この方法はCI/CDパイプラインで利用できる。ただし、本番環境では推奨されない。
```

## 構成の改善

### セクションの分離

````markdown
<!-- 修正前 -->

## 実装方法

URLに認証情報を埋め込む方法は以下の通りである。

\```bash
git clone https://username:password@github.com/user/repo.git
\```

ただし、セキュリティリスクがあるため注意が必要である。

<!-- 修正後 -->

## 実装方法

URLに認証情報を埋め込む方法は以下の通りである。

\```bash
git clone https://username:password@github.com/user/repo.git
\```

## 注意点

セキュリティリスクがあるため、以下に注意が必要である。
````

**理由**: 注意点は独立したセクションに分離。**目安**: 注意点 / 参考リンク / 補足が 2 行以上になる場合は独立セクションに分離する。1 行で収まる場合は本文末尾のインライン (`ただし、〜`) で十分。

## コード例の改善

### コメントの追加

````markdown
<!-- 修正前 -->

```javascript
const result = doSomething();
processData(result);
```

<!-- 修正後 -->

```javascript
// データを取得
const result = doSomething();

// 取得したデータを処理
processData(result);
```
````

### 比較パターンの追加

````markdown
<!-- 修正前 -->

```javascript
const data = getData() ?? defaultData;
```

<!-- 修正後 -->

従来の方法:

```javascript
const data = getData();
if (data === null || data === undefined) {
  data = defaultData;
}
```

改善後:

```javascript
const data = getData() ?? defaultData;
```
````

## 編集の優先順位

1. **簡潔性** - 無駄な文章の徹底的な削除
2. **実用性** - 自分が使うための背景情報は保持
3. **正確性** - 技術的事実の徹底検証
4. **読みやすさ** - 自然で分かりやすい表現

## 編集プロセス

1. **内容の簡潔化** - 冗長表現・AI的表現の削除
2. **構成の最適化** - 見出し・章立ての改善
3. **事実確認** - 外部資料によるファクトチェック
4. **表現の調整** - 文章表現・textlint対応

## 編集時の注意事項

<!-- textlint-disable -->

- 引用・例文として意図的に残された「です・ます」調は変更しない
<!-- textlint-enable -->
- コードブロックには必ず言語指定を追加
- 画像パスは必ず`/images/`で開始
