## 見出しの改善

### 冗長な見出しの簡潔化

```markdown
<!-- 修正前 -->

## なぜURLに認証情報を埋め込むのか

<!-- 修正後 -->

## 課題
```

**理由**: 疑問形は冗長。名詞形で簡潔に。

### 長い見出しの分割

```markdown
<!-- 修正前 -->

## 使用上の注意と推奨場面

<!-- 修正後 -->

## 注意点

## 適用場面
```

**理由**: 複数の内容を1つの見出しに詰め込まない。

### 曖昧な見出しの具体化

```markdown
<!-- 修正前 -->

## Git URLについて

<!-- 修正後 -->

## Git URL形式
```

**理由**: 「について」は曖昧。具体的な名詞で表現。

## 技術的補足の追加

### 用語の説明

```markdown
<!-- 修正前 -->

userinfo部分に認証情報を含める。

<!-- 修正後 -->

userinfo（URI仕様RFC 3986で定義された`username:password`形式の部分）に認証情報を含める。
```

**理由**: 技術用語の初出時は説明を追加。

### セキュリティ注意事項

```markdown
<!-- 修正前 -->

URLに認証情報を埋め込む。

<!-- 修正後 -->

URLに認証情報を埋め込む。ただし、この方法はセキュリティリスクがあるため、ログ記録や履歴に注意が必要である。
```

### 制約条件の明示

```markdown
<!-- 修正前 -->

この方法はCI/CDパイプラインで利用できる。

<!-- 修正後 -->

この方法はCI/CDパイプラインで利用できる。ただし、本番環境では推奨されない。
```

<!-- 修正後 -->

### 比較パターンの追加


## 編集の優先順位

1. **簡潔性** - 無駄な文章の徹底的な削除
2. **実用性** - 自分が使うための背景情報は保持
3. **正確性** - 技術的事実の徹底検証
4. **読みやすさ** - 自然で分かりやすい表現

## 編集プロセス

1. **内容の簡潔化** - 冗長表現・AI的表現の削除
2. **構成の最適化** - 見出し・章立ての改善
3. **事実確認** - 外部資料によるファクトチェック
4. **表現の調整** - 文章表現・textlint対応

## 編集時の注意事項

<!-- textlint-disable -->

- 引用・例文として意図的に残された「です・ます」調は変更しない
<!-- textlint-enable -->
- コードブロックには必ず言語指定を追加
- 画像パスは必ず`/images/`で開始
