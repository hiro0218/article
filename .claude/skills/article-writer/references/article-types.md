# 記事タイプ

`article-writer` がアウトライン・下書き・加筆を行うときに読む。ここでは記事タイプの判定と最小構成だけを定める。詳細テンプレートや編集観点は、記事リポジトリの writing rules や `article-foundation-editor` が存在する場合はそちらへ委譲する。

## タイプ判定

記事タイプを1つ選ぶ。入力オプションは不要である。ユーザーが自然文で「調査記事として」「短いメモで」など方向性を明示した場合は、それを優先する。

| タイプ       | 非対称に出る特徴                                                           | 最小構成                                   |
| ------------ | -------------------------------------------------------------------------- | ------------------------------------------ |
| 問題解決型   | 具体的なエラー、詰まり、移行、実装課題がある                               | 背景、環境、原因、解決方法、注意点、まとめ |
| 調査・比較型 | 主張の検証、ツール比較、実測結果の整理が主目的である                       | 概要、課題、評価基準、結果、考察、参考     |
| 知識共有型   | 概念、API、構文、現在の挙動を説明する                                      | 概要、定義、使い方、注意点、参考           |
| 設計論型     | 設計、作業手順、判断基準について主張がある                                 | 概要、背景、問題、代替案、注意点、まとめ   |
| その他       | 上記に当てはまらない。雑記、レビュー、イベント感想、作品考察、短いメモなど | 内容が伝わる見出しで自由に構成する         |

判定材料が弱い場合は、説明中心なら知識共有型、主張中心なら設計論型に倒す。技術記事として扱う根拠が弱い場合は、else としてその他を選ぶ。

## 章の役割

章ごとに役割を1つ決める。役割がない章は削るか、隣の章へ統合する。

- 問題提起: 具体的な状況や失敗条件を示す。
- 定義: API、手法、概念を使う前に定義する。
- 根拠: 例、一次情報、コード、出力、観測結果を示す。
- 比較: 従来の方法と改善後、選択肢AとB、変更前後を比べる。
- 実装: コマンド、コード、設定、構成を具体的に示す。
- 検証: 環境、結果、出力、再現性の範囲を示す。
- 制約: 適用できない条件や注意点を示す。
- 結論: 新しい主張を足さず、観測結果をまとめる。

## タイプ別の注意

### 問題解決型

- 問題、原因、解決方法の対応関係を崩さない。
- 技術記事であれば、実行可能なコマンドやコード例を優先する。
- 派生エラーや当てはまらない条件も必要に応じて書く。

### 調査・比較型

- 評価基準を先に置く。
- 表は比較が読みやすくなる場合だけ使う。
- 結論は観測結果から言える範囲に絞る。

### 知識共有型

- 用語や API を先に定義する。
- 例は理解に必要な最小限にする。
- 対応バージョンや非対応ケースを明示する。

### 設計論型

- 主張が効く場面と効かない場面を分ける。
- 代替案を公平に扱う。
- 抽象論だけで終わらせず、判断基準を残す。

### その他

- 「なんでもあり」の else として扱う。型に押し込まない。
- 体験、観察、考察、レビュー、感想など、依頼内容に近い形でまとめる。
- 技術記事向けのコード例、検証環境、再現性要件を機械的に要求しない。
- 作品考察やレビューでは、確認できる事実と解釈を分ける。
- 対象範囲やネタバレ許容範囲が未指定で、ユーザーが下書きだけを求めている場合は、仮定を置いて進める。これらは保存時の確認事項であり、不明瞭点として扱わない。

## 確認質問

すでに分かっている内容は聞かない。質問は最大2つまでにする。

- 想定読者は誰か。
- 読者に一番残したい要点は何か。
- 読者から最も反論されそうな点は何か。
- 記事の根拠にする検証、デモ、一次情報は何か。
